# 产品思维

> "从用户需求出发，设计、验证、迭代产品。"

## 核心理念

好的产品不是"做出来的"，而是"长出来的"：
- 发现真实需求
- 设计最小可行方案
- 快速验证
- 持续迭代

## 什么时候使用

- 用户说"设计一个产品"
- 用户说"用户需求是什么"
- 需要定义MVP
- 产品迭代规划

## 工作流程

```
用户需求 → 问题定义 → 解决方案 → MVP → 验证 → 迭代
```

### 步骤详解

**1. 用户需求**
- 目标用户是谁？
- 核心痛点是什么？
- 场景是什么？

**2. 问题定义**
- 一句话描述问题
- 验证这是真问题

**3. 解决方案**
- 核心功能
- 差异化价值

**4. MVP设计**
- 必须有的功能
- 可以砍掉的功能

**5. 验证指标**
- 如何衡量成功？
- 什么情况算失败？

**6. 迭代路线**
- 第一阶段做什么？
- 后续如何演进？

## 产品框架

### Jobs to be Done (JTBD)
- 用户"雇佣"产品来完成什么任务？
- 关注任务，而非产品本身

### 用户故事
```
作为 [角色]
我想要 [功能]
以便于 [目的]
```

### 优先级矩阵
```
        紧急
         ↑
    P1   │   P0
重要 ─────┼───── 不重要
    P2   │   P3
         ↓
        不紧急
```

### 产品生命周期
```
引入期 → 成长期 → 成熟期 → 衰退期
```

## 经典案例

### 案例：微信红包

**用户需求**：
- 春节发红包是传统
- 但线下发红包麻烦

**问题定义**：
- 如何让发红包更方便？

**MVP**：
- 发红包、抢红包
- 简单动画

**验证**：
- 2014春节，800万用户使用

**迭代**：
- V2：红包记录
- V3：群红包、拼手气

## 注意事项

1. **避免伪需求**
   - 用户说的 ≠ 用户要的
   - 要观察行为，不只听说法

2. **MVP要够小**
   - 不是"功能少的产品"
   - 是"刚好能验证假设的最小产品"

3. **快速验证**
   - 不要憋大招
   - 尽早面对用户

4. **数据驱动**
   - 用数据验证假设
   - 不要凭感觉决策

5. **持续迭代**
   - 产品永远没有"完成"
   - 根据反馈持续改进
